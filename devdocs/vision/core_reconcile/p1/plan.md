# Phase 1 Implementation Plan: dnsmasq workflow in nctl, retire the Job-export path

Parent: [roadmap.md](../roadmap.md) ‚Äî Phase 1: move dnsmasq rendering into `nctl_core`, make
`ansible_agdev` deploy-only, and delete the old Nautobot-Job export path.

## Current state (as of 2026-07-14)

- Phase 0-EX1 is complete ([p0-ex1/report2-4.md](../p0-ex1/report2-4.md)): all desired-state
  models incl. `IntentEvaluation` are queryable at `/api/graphql/`, with pinned snake_case names
  (`desired_endpoints`, `desired_ip_ranges`, `intent_evaluations`, relation
  `DesiredNodeType.desired_endpoints`, JSONFields as `GenericScalar` returning native JSON).
- Rendering today lives in `nintent`:
  - `nautobot_intent_catalog/dnsmasq.py` ‚Äî pure functions (`export_dnsmasq_records`,
    `resolve_dhcp_reservation`, `render_dnsmasq_records_conf`, `render_dnsmasq_export_json`),
    schema_version `3.0`, deterministic sort orders and a fixed `skip_reasons` vocabulary.
    They read inputs via `getattr` on ORM objects.
  - `jobs.py::ExportDnsmasqRecords` ‚Äî feeds those functions from the ORM and publishes
    `dnsmasq-records.conf` + `dnsmasq-export.json` as JobResult files. **Verified: this Job is
    `dnsmasq.py`'s only consumer** (other `dnsmasq` grep hits in `evaluations.py` are model
    fields like `generate_dnsmasq`, unaffected).
- Render inputs beyond `DesiredEndpoint`/`DesiredIPRange`: the DHCP-reservation MAC comes from
  the **latest `IntentEvaluation` per target** ‚Äî `jobs.py::_latest_evaluations(target_type)`
  filters by `target_type` in `{"desired_endpoint", "desired_node"}`, orders by
  `-reviewed_at, -created`, keeps the first row per `target_id`. Endpoint evaluations supply
  `observed_facts.dhcp_mac_candidates` and `deterministic_summary.dhcp_reservation_ready`;
  node evaluations supply `actual_refs`.
- `ansible_agdev/playbooks/nautobot/deploy_nintent_dnsmasq_records.yml` is two plays:
  1. ~260 lines of Job-run / JobResult-polling / file-proxy-regex plumbing that downloads the
     conf to `/tmp/ansible-nintent-dnsmasq/dnsmasq-records.conf` and asserts schema `3.0`;
  2. a deploy play (`hosts: dnsmasq_server`) that copies the conf to
     `/etc/dnsmasq.d/nintent-records.conf` with `validate: dnsmasq --test` and a restart
     handler. Play 2 is the part worth keeping.
  The shared task files `tasks/nautobot_run_job.yml` / `tasks/nautobot_download_file.yml` are
  used by the other export playbooks and stay.
- The `dnsmasq_server` group comes from the generated inventory
  (`inventories/generated/hosts_intent.yml`, produced by `export_nintent_hosts_intent.yml`) ‚Äî
  an implicit prerequisite of any apply.
- nctl (Phase 0): `Config` (`nctl.toml`: `[nautobot]`, `[inventory]`, `[events]`, `[repo]`),
  `NautobotClient.graphql()`, output envelope (`emit`), JSON Lines events + operation IDs
  (`events.py`), Typer CLI with `status` only.
- Deployment constraint (`.local/localenv_memo.md`): dev Nautobot installs nintent from GitHub ‚Äî
  verifying nintent changes costs a push + `docker compose build --no-cache` + restart cycle.
  Plan sequences the work so nintent needs **one** push cycle, and only after parity is proven.

## Approach

Port the renderer as pure functions into `nctl_core`, prove **output parity** against the live
Job before deleting anything, then cut over: deploy-only playbook, `render`/`apply` commands,
delete the Job path. The parity gate is what makes the breaking change safe without keeping two
renderers alive.

Ordering rationale: Steps 1‚Äì3 (nctl renderer + `render` command + parity check) touch only nctl.
Step 4 (nintent deletion) is the single push cycle. Steps 5‚Äì6 (playbook + `apply`) land after
the new source of truth is proven. The skill and report close it out.

**Risk to verify first (start of Step 2)**: exact GraphQL field availability/filterability of
`intent_evaluations` (`target_type` filter, `target_id`, `reviewed_at`, `created`,
`observed_facts`, `deterministic_summary`, `actual_refs`) and `desired_ip_ranges`
(`start_address`, `end_address`, `dnsmasq_options`, `range_policy`, `lifecycle`,
`generate_dnsmasq`) on the live instance. p0-ex1 pinned the type names but exercised only
node/endpoint fields. Resolve empirically before pinning the query.

## Step 1 ‚Äî Port the renderer into `nctl_core` (pure, mapping-based)

- New module `nctl/src/nctl_core/dnsmasq.py`: port `export_dnsmasq_records`,
  `resolve_dhcp_reservation`, `_dns_record_entry`, `_dhcp_range_entry`, skip-reason helpers,
  `render_dnsmasq_records_conf`, and the JSON payload builder from
  `nintent/nautobot_intent_catalog/dnsmasq.py`.
  - Adapt input access from `getattr`-on-ORM to plain mappings (the shape GraphQL returns).
    Everything else ‚Äî record lines, sort keys, `skip_reasons` vocabulary, summary counts ‚Äî
    stays byte-identical.
  - Header changes: `# Generated by nctl` instead of the Job banner; `# operation_id: <id>`
    replaces `# job_result_id`; bump `DNSMASQ_EXPORT_SCHEMA_VERSION` to `4.0` (the generator
    changed; nothing asserts on it anymore, but keep the header honest for humans and Phase 2).
- Port `nintent`'s `tests/test_dnsmasq.py` cases to pytest with dict fixtures (same expected
  lines/skip reasons), so the vocabulary survives the move.

## Step 2 ‚Äî GraphQL fetch layer for the renderer

- After the empirical field check (risk note above), pin one query in
  `nctl_core/dnsmasq.py` (or a small `queries.py`) fetching, in a single request:
  - `desired_endpoints` with `desired_node { id name slug lifecycle }` and all fields the
    renderer reads (`name`, `endpoint_type`, `ip_address`, `ip_policy`, `dns_name`,
    `mdns_name`, `vpn_dns_name`, `generate_dnsmasq`, `dnsmasq_record_type`, `id`),
  - `desired_ip_ranges` with the range fields above,
  - `intent_evaluations` (filtered to the two target types if the FilterSet allows; otherwise
    fetch and split client-side) with `target_type`, `target_id`, `reviewed_at`, `created`,
    `observed_facts`, `deterministic_summary`, `actual_refs`.
- Replicate `_latest_evaluations` client-side: group by `target_id`, sort
  `-reviewed_at, -created`, keep first. Unit-test this selection against tie cases.
- If GraphQL turns out to mishandle a needed field (e.g. an unfilterable evaluation set at
  scale), the fallback is fetching that one collection via the existing REST ViewSet ‚Äî note the
  deviation in the report rather than blocking the phase. Reads stay in `nctl_core` either way.

## Step 3 ‚Äî `nctl render dnsmasq` + parity gate

- CLI: `nctl render dnsmasq [--out PATH] [--json]`.
  - Default: print the conf to stdout (pipeable). `--out` writes the file and prints a summary.
  - `--json`: envelope `nctl.render.dnsmasq.v1` ‚Äî `data` = the export payload (summary,
    dns_records, dhcp_reservations, dhcp_ranges, skipped, conf text, schema_version). This
    payload is deliberately the Phase 2 "desired side" input shape.
  - Render is fast/synchronous: no operation ID or event log for `render` itself.
- **Parity gate (before Step 4 deletes anything)**: run the live `Export dnsmasq Records` Job
  once more, download `dnsmasq-records.conf` + `dnsmasq-export.json`, and diff against
  `nctl render dnsmasq` output on the same live data ‚Äî record lines must match exactly
  (headers/`generated_at`/ids excluded), summary counts and skip reasons must match. Save both
  artifacts as fixtures in the report directory. Any mismatch is a Step 1/2 bug to fix first.

## Step 4 ‚Äî Delete the Job-export path in nintent (single push cycle)

- Delete `ExportDnsmasqRecords` from `jobs.py` (and the line-24 import),
  `dnsmasq.py`, and `tests/test_dnsmasq.py`. `_latest_evaluations` stays (other Jobs use it).
- Bump nintent to 0.5.0; run `uv run python3 -m unittest discover` locally.
- Commit, ask the user to push, rebuild dev Nautobot
  (`docker compose --env-file ../.env build --no-cache && ... up -d`), verify the Job list no
  longer shows "Export dnsmasq Records" and `nctl status` stays green.

## Step 5 ‚Äî Deploy-only playbook in ansible_agdev

- Rewrite `playbooks/nautobot/deploy_nintent_dnsmasq_records.yml` down to (a renamed)
  `playbooks/dnsmasq/deploy_dnsmasq_records.yml`: only the former play 2.
  - Required var `dnsmasq_records_src` (absolute path to a pre-rendered conf); fail fast with a
    clear message if unset or missing. Keep the Linux assert, `validate: dnsmasq --test`, the
    restart handler, and the `/etc/dnsmasq.d/nintent-records.conf` destination.
  - Delete the export play entirely: Job lookup, polling, file-proxy regex matching, and the
    schema `3.0` assert (generator and consumer are now the same tool; the header is
    informational).
- It moves out of `playbooks/nautobot/` because it no longer talks to Nautobot at all.

## Step 6 ‚Äî `nctl apply dnsmasq`

- CLI: `nctl apply dnsmasq [--yes] [--json]`. Flow:
  1. render (Steps 1‚Äì2) to a staging file under the operation's event/artifact directory;
  2. run `ansible-playbook` on the deploy-only playbook with
     `-e dnsmasq_records_src=<staging>`, in `--check --diff` mode by default;
  3. with `--yes`, run again without `--check` after the dry-run (or directly when the caller
     already reviewed ‚Äî keep it simple: `--yes` skips straight to the real run).
- Config: new `[ansible]` table in `nctl.toml` ‚Äî `playbook_dir` (ansible_agdev checkout),
  `inventory` (path to `inventories/generated/hosts_intent.yml` or the inventory dir), plus the
  strict-model treatment existing sections get. `apply` errors out with a pointed message if
  the generated inventory is missing ("run export_nintent_hosts_intent.yml first") ‚Äî migrating
  that export is explicitly out of scope.
- This is the first long-running command: mint an operation ID, emit JSON Lines events
  (`started`, `rendered`, `dry_run_completed`, `apply_started`, `apply_completed`, `failed`)
  via the Phase 0 `events.py` layer, and include ansible's exit code/summary in the final
  envelope (`nctl.apply.dnsmasq.v1`). Exit code: 0 clean, 1 failure ‚Äî dry-run with pending
  changes is still 0 (the diff is the deliverable, not an error).
- Tests: unit-test the command orchestration with the ansible invocation faked (respx for
  GraphQL, a stub `ansible-playbook` on PATH or subprocess seam); don't try to run real ansible
  in pytest.

## Step 7 ‚Äî Skill, docs, report

- `.claude/skills/update-dnsmasq/SKILL.md` (parent repo): "update dnsmasq" resolves to
  `nctl apply dnsmasq` (review diff) ‚Üí `nctl apply dnsmasq --yes`, with `nctl render dnsmasq`
  for inspect-only. Note the inventory prerequisite.
- `nctl/README.md`: document the two commands and the `[ansible]` config table.
- `ansible_agdev` README (if it documents the old playbook): update the dnsmasq section.
- Write `devdocs/vision/core_reconcile/p1/report*.md` in the established style, including the
  parity artifacts (Step 3), the pinned GraphQL query, and any fallbacks taken.

## Out of scope

- Migrating `Export Ansible Hosts Intent` / `Export Production Inventory` /
  `Sync Deployment Profiles` to nctl (same principle, later phase ‚Äî recorded in the roadmap).
- Drift computation (`nctl drift`, Phase 2) ‚Äî but the render payload shape is designed for it.
- The bootstrap dnsmasq role (`roles/dnsmasq_server`, `playbooks/bootstrap/setup_dnsmasq.yml`)
  ‚Äî it installs the service; this phase only manages the records conf.
- Any change to `IntentEvaluation` production (the Evaluate Jobs stay as the MAC/actual-refs
  source; replacing them with direct nodeutils reads is Phase 2 territory).

## Exit criteria (from roadmap, made checkable)

- [ ] `nctl render dnsmasq` output matches the last Job export on live data (record lines,
  summary counts, skip reasons ‚Äî parity artifacts saved in the report).
- [ ] `nctl apply dnsmasq` shows an ansible `--check --diff` dry-run by default and applies
  with `--yes`, emitting an operation ID and JSON Lines events; verified against the dev
  cluster end to end.
- [ ] `nintent` contains no dnsmasq rendering: `ExportDnsmasqRecords`, `dnsmasq.py`, and
  `test_dnsmasq.py` are deleted; nintent's test suite passes; the dev Nautobot Job list no
  longer offers the export.
- [ ] The dnsmasq playbook is deploy-only (no Job/file-proxy/polling tasks) and takes
  `dnsmasq_records_src` as input.
- [ ] `.claude/skills/update-dnsmasq` exists and resolves to the two-command sequence.
- [ ] `uv run pytest` passes in nctl, including the ported dnsmasq vocabulary tests.

## Suggested commit order

1. nctl: renderer port + GraphQL fetch + `render dnsmasq` + tests (Steps 1‚Äì3; parity gate run
   against live before proceeding).
2. nintent: delete the Job-export path, version bump (Step 4; the single push/rebuild cycle).
3. ansible_agdev: deploy-only playbook rewrite (Step 5).
4. nctl: `apply dnsmasq` + `[ansible]` config + events + tests (Step 6).
5. Parent repo: skill, submodule pointer bumps, `p1/report*.md` (Step 7).
