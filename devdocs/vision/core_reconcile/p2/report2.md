# Phase 2 Report — Step 2 (Production composer port, `nctl render production`, parity gate)

Date: 2026-07-15. Implements [p2/plan.md](plan.md) Step 2.

## What was built

New package `nctl/src/nctl_core/production/` plus one top-level orchestration module:

- `contract.py` — ported from nintent's `production_inventory_contract.py`: `PRODUCTION_INVENTORY_SCHEMA_VERSION`,
  `ContractError`, `canonical_json`/`canonical_json_digest`, `validate_deployment_profiles`,
  `map_placement_config`, `validate_endpoint_ownership`, `evaluate_platform_policy`,
  `actual_state_problem`, `resolve_connection_variables`, `merge_host_variables`,
  `validate_production_inventory_document`, `validate_production_report`, and their private
  helpers — unchanged logic. **Not ported** (per the plan's Decision on the byte contract):
  `parse_profile_job_input` and its `_raise_invalid_constant` helper (the Job-input transport),
  and `validate_desired_service_reference`/`validate_endpoint_reference`/`require_unique_reference`
  (used only by nintent's YAML-catalog importer, which stays ledger-side and isn't part of
  production composition — confirmed by grep before dropping them).
- `composer.py` — ported from `production_inventory.py`: the `EndpointInput`/
  `OperationalConfigInput`/`PlacementInput`/`RealizedState`/`NodeInput`/`ProductionComposition`
  dataclasses, `is_production_eligible`, `compose_production_inventory`,
  `render_production_inventory_yml`, `render_production_report_json`, and their private
  helpers — unchanged logic, except the inventory header line
  (`# Generated by nctl render production` instead of `# Generated by Nautobot Intent Catalog
  production inventory composer`, same header-change convention as Phase 1's dnsmasq renderer).
  Imports `ActualFacts`/`read_actual_facts`/`actual_type_problem`/`missing_required_facts` from
  `nctl_core.sources.actual` (already ported there in Step 1) instead of re-porting them.
- `profiles.py` — reads `<ansible.playbook_dir>/vars/deployment_profiles.yml` directly,
  extracts the `deployment_profiles` key, validates it with the ported `validate_deployment_profiles`,
  and computes `canonical_json_digest` locally. Replaces nintent's Job-input byte contract
  transport (`parse_profile_job_input`, `verify_deployment_profiles_contract.yml`,
  `nintent_serialize_deployment_profiles.yml`) — that machinery existed only because composition
  ran inside Nautobot while the profiles lived in ansible_agdev; now that nctl can read the
  ansible_agdev checkout directly (`[ansible] playbook_dir` in `nctl.toml`, already configured
  since Phase 1), the transport half has no reason to exist. The digest is still computed and
  carried through the schema 1.0 document/report as provenance.
- `adapter.py` — `build_production_node_inputs(snapshot)`, mirroring nintent's
  `jobs.py::_build_production_node_inputs` but reading Step 1's typed `SourceSnapshot` instead of
  the ORM. `DesiredNodeOperationalConfig.local_endpoint`/`tailscale_endpoint` are already shaped
  like `EndpointInput` (ported by the same Step 1 fetch layer), so mapping them across is a
  field-for-field copy, not a second query.
- `nctl_core/production_render.py` — `build_production_render(cfg)` (fetch + compose, one
  synchronous call, same "no operation ID for renders" convention as `dnsmasq_render.py`) and
  `write_production_artifacts(envelope, out_dir)`: stages the rendered YAML, validates it with
  `ansible-inventory --list`, then atomically replaces `<out_dir>/production.yml` and writes
  `<out_dir>/production.reports/<generation_id>.json`. Never leaves a partial/unvalidated file
  behind on failure (verified by test, see below).
- CLI: `nctl render production [--out DIR] [--json]` wired into `cli/main.py`. Without `--out`,
  the inventory YAML goes to stdout (pipeable, matches `render dnsmasq`'s default). With
  `--out DIR`, writes the two artifacts and prints a summary instead.

## Deviation from plan: `--out` semantics

The plan's Step 2 CLI sketch says `--out` "(default: the configured generated-inventory dir)" —
read as: passing `--out` with no argument should fall back to `cfg.ansible.inventory`'s
directory. Typer/Click do not cleanly support an optional-value-with-fallback for a `Path`
option without a `nargs`-hack that would add real complexity for a render command. Implemented
instead: `--out DIR` always requires an explicit directory (pass the directory containing the
configured `ansible.inventory` path to regenerate it in place). This keeps the flag's behavior
simple and explicit; documented in the option's `--help` text. No functional capability is
lost — it's a CLI-ergonomics call, not a feature gap.

## Live parity gate against the real `Export Production Inventory` Job

Ran both the ported composer and the actual live nintent Job against the same live dev Nautobot
data (`http://localhost:8000`), rather than relying on the dataset-shape prediction from
report1.

**Real bug found in `ansible_agdev` while setting this up** (unrelated to this phase's port,
noted for completeness since it blocked a naive parity run): `playbooks/nautobot/export_nintent_production.yml`
declares `vars_files: [../vars/nautobot.yml, ../vars/deployment_profiles.yml]`, but those
paths are relative to the playbook's own directory (`playbooks/nautobot/`), which resolves to
the nonexistent `playbooks/vars/` — the real `vars/` directory is two levels up
(`ansible_agdev/vars/`). Confirmed by `ls playbooks/nautobot/../vars/` failing and by the Job
run failing with `'deployment_profiles' is undefined` on first attempt (the `nautobot_url`/
`nautobot_token` vars load-failure was masked by their Jinja `default(..., true)` fallback to
env vars, so only the profile var's absence surfaced). Worked around for this one-off diagnostic
run with `ansible-playbook ... -e @vars/deployment_profiles.yml` (no playbook edit); did not fix
the playbook itself since it is deleted wholesale in this phase's Step 7 and touching it now is
out of scope. Recording this here so Step 7 doesn't need to rediscover it.

With that workaround, the live Job ran to completion (`job_result_id:
13fde479-9254-4e18-895e-3575974edc7b`, `generation_id: 7aec82c2-0635-47e3-b0b4-a7c11fd1552e`) and
`nctl render production --out` ran against the same live data (`generation_id:
c364bd12-852a-48b4-8503-97094626b94c`). Diffed both `production.yml` and the companion report
with generation-id/generated-at/report-path fields excluded (per the plan's parity-gate rule):

- **Inventory YAML**: identical except the intentional header line and the (correctly
  generation-id-derived) `nintent_report_path`. All five core groups, host lists, and `vars`
  match exactly — both empty (`ssh_hosts: {}`, etc.).
- **Deployment profile digest**: `d3c8ce79a70668062ef71d9f78c5886f6ba088ad731f66cb1f466100bf16d638`
  on both sides — confirms nctl's Python `canonical_json`/SHA-256 computation over
  `vars/deployment_profiles.yml` produces byte-identical output to Ansible's
  `to_json(sort_keys=true, ...) | hash('sha256')` serialization of the same file.
  the same profile file.
- **Report JSON**: byte-identical after excluding `generation_id`/`generated_at`/
  `deployment_profile_digest`/`report_path` (verified programmatically, `MATCH`).

As predicted in report1, the current dev dataset has zero production-eligible nodes (all
`desired_nodes` are `lifecycle: planned`; zero `desired_node_operational_configs`), so both runs
produced the empty-but-well-formed inventory (`summary.eligible: 0`). This is still a valid
parity result: the composer's *shape and behavior* on identical zero-input data are proven
identical, which is what Step 2 is actually testing (the composition logic, not the dataset).
Populating richer test data to exercise non-empty composition is outside this phase's scope —
the ported pytest suite (below) already covers the full behavior matrix with synthetic
`NodeInput`s.

`nctl render production --out <dir>` was also run standalone end-to-end (writing
`production.yml` + `production.reports/<id>.json`, with the real `ansible-inventory --list`
validation step actually executing and passing) to confirm the write path works outside the
parity comparison, not just the fetch+compose path.

## Tests

- `tests/test_production_composer.py` — full port of nintent's
  `tests/test_production_inventory.py` (unittest → pytest), using
  `nctl_core.sources.actual.ActualFacts` in place of nintent's dataclass. Covers: actual-backed
  Linux/macOS composition, observed-vs-expected OS drift, WoL power-managed grouping, tailscale
  connection resolution, declared HAOS composition (with and without a service placement),
  mixed-platform pipelines, the full skip taxonomy (missing/unsupported/stale actual data,
  missing MAC for WoL), dangling-group avoidance, all four global-failure `ContractError` paths,
  multi-service/multi-instance placement composition, disabled-placement exclusion, lifecycle/
  node-type eligibility filtering, byte-stable determinism, and schema-versioned renderers.
- `tests/test_production_contract.py` — the subset of nintent's
  `tests/test_production_inventory_contract.py` covering ported functions only (canonical JSON,
  profile-shape validation and duplicate-variable rejection, placement config mapping,
  connection resolution incl. the `unresolved_connection_path` error, the 72-hour freshness
  boundary, all platform-policy scenarios incl. drift and the invalid-power error, endpoint-
  ownership mismatch, host-variable merge conflict, and closed inventory/report schema
  validation).
- `tests/test_production_profiles.py` — `load_deployment_profiles` against a written
  `vars/deployment_profiles.yml`: valid load + digest, missing file, missing top-level key,
  invalid profile shape (each surfaced as `DeploymentProfilesError`).
- `tests/test_production_adapter.py` — `build_production_node_inputs` joins operational config,
  placements, and realized-device actual facts correctly from a `SourceSnapshot`; handles a node
  with no operational config/placements; handles a realized VM (no device lookup); sorts by
  slug.
- `tests/test_production_render.py` — `write_production_artifacts`: propagates a render failure
  without touching the filesystem, requires `ansible-inventory` on PATH, atomically replaces on
  success (verified no leftover staging file), and leaves no file behind when
  `ansible-inventory --list` rejects the staged copy.
- `tests/test_cli_render_production.py` — default stdout rendering, `--json` envelope, `--out`
  writing artifacts and printing a summary, `--out` propagating a write error, and exit code 1
  on render failure — mirrors the existing `test_cli_render_dnsmasq.py` pattern.
