# Report — Step 3: deterministic dnsmasq bytes and a desired digest

Date: 2026-07-22
Scope: `nctl` (submodule)
Status: **complete** (focused: `test_dnsmasq.py` + `test_dnsmasq_render.py` +

## Goal (plan.md Step 3)

Make the deployed dnsmasq conf byte-deterministic in source state alone --
today's `generated_at`/`operation_id` comments mean re-rendering unchanged
desired state produces a different file every time, which would make Step
5's content-drift comparison detect false drift on every reconcile round.
Add one pure `dnsmasq_content_sha256` digest helper and a shared
render-from-snapshot implementation Step 5's drift computation will reuse.

## Changes

### 1. `nctl_core/dnsmasq.py`
- `DNSMASQ_EXPORT_SCHEMA_VERSION` bumped `"4.0"` -> `"5.0"` (deliberate
  breaking byte-contract change).
- `render_dnsmasq_records_conf(export)` no longer takes `generated_at`/
  `operation_id` and no longer emits `# generated_at: ...`/`# operation_id:
  ...` comment lines -- the deployed bytes are now exactly:
  ```
  # Generated by nctl
  # schema_version: 5.0
  <sorted DNS/DHCP directives>
  ```
  (directives were already sorted by `export_dnsmasq_records`; no change
  there.) `dnsmasq_export_payload`/`render_dnsmasq_export_json` (the JSON
  envelope) are unchanged -- `generated_at`/`operation_id` remain there and
  in the operation event log/artifact metadata only.
- Added `dnsmasq_content_sha256(conf: str) -> str`: lowercase hex SHA-256 of
  the exact UTF-8 conf bytes, the one standard full-file digest algorithm
  (no comment-stripping, no second canonicalizer) nodeutils will reproduce
  independently in Step 4.

### 2. `nctl_core/dnsmasq_render.py`
- `RENDER_DNSMASQ_SCHEMA` bumped `nctl.render.dnsmasq.v1` -> `v2`.
- `DnsmasqRenderData` gained `content_sha256: str`.
- Added `DnsmasqRenderResult` (`export`, `conf`, `content_sha256`) and
  `compute_dnsmasq_render(snapshot: SourceSnapshot) -> DnsmasqRenderResult`:
  the one pure, I/O-free implementation from `SourceSnapshot` to rendered
  conf + digest. `build_dnsmasq_render` (the real Nautobot-fetching CLI/apply
  path) now calls it instead of duplicating the
  fetch-inputs/export/render sequence inline. Step 5's drift comparator will
  call the same function against the snapshot drift already fetched, so "what
  would `nctl render dnsmasq` produce right now" can never independently
  disagree between the two call sites.

### 3. `nctl_core/dnsmasq_apply.py`
- `APPLY_DNSMASQ_SCHEMA` bumped `nctl.apply.dnsmasq.v1` -> `v2`.
- `DnsmasqApplyData` gained `content_sha256: str`, populated from
  `render.data.content_sha256` and included in the `"rendered"` operation
  event.

## Test changes

- `tests/test_dnsmasq.py`: updated the two `render_dnsmasq_records_conf`
  call sites for the dropped `generated_at`/`operation_id` params and the
  `4.0` -> `5.0` schema string. Added 5 new tests: equal source state at
  different times/operation IDs renders byte-identical conf and digest (and
  `dnsmasq_export_payload`'s `dns_records` are equal too); a changed IP
  address changes the digest; a newly added DHCP range changes the digest;
  deployed conf bytes never contain `generated_at`/`operation_id`; the
  digest matches a standard independent `hashlib.sha256` computation over
  the same bytes (and is lowercase hex, 64 chars).
- `tests/test_dnsmasq_render.py`: schema string and `schema_version`
  updated to v2/5.0; `content_sha256` length asserted; the operation-id-
  embedding test rewritten to assert operation_id/`generated_at` are
  **excluded** from conf bytes and the conf is exactly the two-line header
  for empty desired state; envelope JSON key-set assertion includes
  `content_sha256`.
- `tests/test_dnsmasq_apply.py`: `_render()` fixture updated to the new
  two-comment-line conf and schema v2, computing `content_sha256` via the
  real helper. The dry-run and apply tests both now assert
  `envelope.data.content_sha256 == dnsmasq_content_sha256(artifact.read_text())`
  -- what got written to disk is exactly the bytes whose digest was
  reported -- and the dry-run test's stale `# operation_id: ...` assertion
  was replaced with "the operation ID never appears in the artifact".
- `tests/test_cli_render_dnsmasq.py`, `tests/test_cli_apply_dnsmasq.py`:
  fixture schema strings bumped to v2 for consistency (these tests build
  their own fake envelopes and don't depend on the real constant, so this
  is cosmetic, not required for passing).
- `tests/test_compatibility_snapshots.py`: `nctl.apply.dnsmasq.v1` ->
  `v2` (added `content_sha256` to its frozen field set) and
  `nctl.render.dnsmasq.v1` -> `v2` (same) -- replaced, not kept alongside
  the old key, matching the plan's "old compatibility snapshots are
  replaced, not conditionally accepted."

## Handoff to Step 4

- Step 4 (`nodeutils`, `ansible_agdev`, `nctl`, `nauto`) needs the exact
  managed-file path (`/etc/dnsmasq.d/nintent-records.conf` per the plan) as
  validated deployment-profile reconciliation metadata, and must compute the
  same standard full-file SHA-256 `dnsmasq_content_sha256` computes here --
  no comment-stripping or second canonicalizer on the nodeutils side either.
- Step 5's drift comparator should call `dnsmasq_render.compute_dnsmasq_render(snapshot)`
  (built in this step) to get `content_sha256`, then compare it against the
  nodeutils-observed digest Step 4 wires up -- no new render implementation
  should be needed there.
- Per the plan's coordinated-rollout ordering, this nctl-side schema bump
  (`DNSMASQ_EXPORT_SCHEMA_VERSION`, `nctl.render.dnsmasq.v2`,
  `nctl.apply.dnsmasq.v2`) is committed now but is **not yet deployed**
  against the real `agdnsmasq` -- the live `nctl apply dnsmasq`/`reconcile`
  path already builds and would deploy 5.0-schema bytes the next time it
  runs against the real host, which is expected and safe (the plan's
  rollback notes: "the first v5 dnsmasq deployment removes volatile comments
  but preserves the generated directives").
