# nintent dnsmasq Export to ansible_agdev Plan

## Decision

Use multiple dnsmasq configuration files.

- `/etc/dnsmasq.d/ansible.conf`
  - Owned by the existing `dnsmasq_server` role.
  - Keeps daemon-level settings such as listen addresses, interfaces, upstream
    resolvers, local zones, cache, and DHCP options.
- `/etc/dnsmasq.d/nintent-records.conf`
  - Owned by the new Nautobot/nintent export deployment flow.
  - Contains only records generated by nintent's `Export dnsmasq Records` Job.

This keeps Nautobot/nintent as the single source of truth for DNS records
without making nintent responsible for dnsmasq service configuration.

## Current State

### nintent

- `Export dnsmasq Records` creates two JobResult files:
  - `dnsmasq-records.conf`
  - `dnsmasq-export.json`
- The `.conf` output is already dnsmasq-ready and contains lines such as:
  - `host-record=<dns_name>,<ip>`
  - `address=/<dns_name>/<ip>`
  - `cname=<vpn_dns_name>,<dns_name>`
- The Job uses Nautobot `Job.create_file()`, so output files are exposed as
  File Proxy objects and can be downloaded through the Nautobot REST API.

### ansible_agdev

- `roles/dnsmasq_server` already installs dnsmasq and writes
  `/etc/dnsmasq.d/ansible.conf`.
- `playbooks/collect_nodeutils_and_ingest_nautobot.yml` already demonstrates
  the local pattern for:
  - looking up a Nautobot Job by name,
  - running it through the REST API,
  - passing `nautobot_url`, `nautobot_token`, and `nautobot_validate_certs`.

## Target Flow

1. Run nintent's Nautobot Job `Export dnsmasq Records`.
2. Wait until the created JobResult reaches a terminal state.
3. Find the JobResult output file named `dnsmasq-records.conf`.
4. Download that file from the Nautobot File Proxy API.
5. Stage it on the Ansible control node.
6. Copy it to dnsmasq servers as:
   - `/etc/dnsmasq.d/nintent-records.conf`
7. Validate dnsmasq configuration before accepting the update.
8. Restart or reload dnsmasq only when the deployed file changes and validation
   succeeds.

## Proposed Ansible Changes

### 1. Add a deployment playbook

Create:

```text
ansible_agdev/playbooks/deploy_nintent_dnsmasq_records.yml
```

Suggested play structure:

1. `localhost`
   - Assert required Nautobot variables.
   - Look up Job named `Export dnsmasq Records`.
   - Run the Job with `include_skipped: true` or configurable default.
   - Poll the JobResult endpoint until success/failure.
   - Query File Proxy objects for the completed JobResult.
   - Select `dnsmasq-records.conf`.
   - Download it to a local staging path such as:
     - `/tmp/ansible-nintent-dnsmasq/dnsmasq-records.conf`

2. `dnsmasq_server`
   - Ensure `/etc/dnsmasq.d` exists.
   - Copy the staged file to:
     - `/etc/dnsmasq.d/nintent-records.conf`
   - Validate with a full dnsmasq config test.
   - Notify the existing `Restart dnsmasq` handler or perform a reload/restart.

### 2. Add defaults or variables

Likely variables:

```yaml
nintent_dnsmasq_job_name: "Export dnsmasq Records"
nintent_dnsmasq_include_skipped: true
nintent_dnsmasq_local_dir: /tmp/ansible-nintent-dnsmasq
nintent_dnsmasq_local_conf: "{{ nintent_dnsmasq_local_dir }}/dnsmasq-records.conf"
nintent_dnsmasq_config_file: "{{ dnsmasq_config_dir | default('/etc/dnsmasq.d') }}/nintent-records.conf"
nintent_dnsmasq_poll_retries: 30
nintent_dnsmasq_poll_delay: 2
nautobot_validate_certs: true
```

Continue using the existing variables:

```yaml
nautobot_url: https://nautobot.example.local
nautobot_token: "{{ vault_nautobot_token }}"
```

### 3. Keep record variables empty by convention

For Nautobot Single SOT operation, inventory should avoid setting static record
lists in `ansible.conf`:

```yaml
dnsmasq_address_records: []
dnsmasq_cname_records: []
dnsmasq_host_records: []
```

The existing role can still support these variables for bootstrap or emergency
override, but the normal path should treat nintent as the owner of DNS records.

## API Notes

Nautobot 2.1+ supports Job output files created by `Job.create_file()`.

Important API requirements:

- The token must be able to run the `Export dnsmasq Records` Job.
- The token must be able to view/download `extras.fileproxy` objects.
- File download endpoint format is expected to be:
  - `/api/extras/file-proxies/<id>/download/`

The exact File Proxy listing filters should be confirmed against the deployed
Nautobot version. If filtering by `job_result` is available, use it directly.
Otherwise, fetch the JobResult detail and/or query recent File Proxy objects and
select by both JobResult id and file name.

## Validation Strategy

Prefer an atomic-ish update:

1. Copy the downloaded content to a temporary file on the dnsmasq host.
2. Run `dnsmasq --test` against the effective config.
3. Move into place only after validation passes.
4. Restart/reload dnsmasq only after the final file changes.

If Ansible `copy validate` can validate the final candidate in context, use it.
If `dnsmasq --test --conf-file=%s` only validates that single file and misses
interactions with `ansible.conf`, use an explicit command after staging instead.

## Failure Behavior

- If the Nautobot Job fails, do not touch the remote dnsmasq records file.
- If the File Proxy cannot be found or downloaded, do not touch the remote file.
- If dnsmasq validation fails, leave the last known good
  `/etc/dnsmasq.d/nintent-records.conf` in place and fail the play.
- If the export produces zero records, still deploy the generated file if the
  Job succeeded. Empty desired DNS state should be explicit, not silently
  treated as an error.

## Test / Verification Checklist

- Run Ansible syntax check for the new playbook.
- Run the playbook in check mode where practical.
- Test against a real or dev Nautobot instance:
  - Job lookup succeeds.
  - JobResult polling reaches success.
  - `dnsmasq-records.conf` downloads correctly.
  - File copies to the dnsmasq host.
  - dnsmasq validation passes.
  - service restarts only when file content changes.
- Confirm failure cases preserve the previous remote records file.

## Open Questions

- Which Nautobot version is deployed? The plan assumes 2.1+ because Job file
  output depends on `Job.create_file()`.
- Does the deployed API support filtering File Proxy objects by JobResult?
- Should dnsmasq be restarted or reloaded? The existing role uses restart, so
  reuse restart unless there is a local reason to prefer reload.
- Should `dnsmasq-export.json` also be archived locally for audit/debugging?
